# Bounded Context: Security Operations

> *Core subdomain.* Detection, triage and lifecycle management of
> security findings. The customer-facing reason NOIP exists.

## Purpose

Detect security-relevant issues against discovered infrastructure
(secrets, vulnerabilities, integrity violations) and against NOIP's
own audit stream (auth anomalies, brute force). Manage their
lifecycle from raised → acknowledged → resolved (or suppressed).

## Ubiquitous language (canonical)

`Security Event` · `Severity` · `Finding` · `Secret (finding)` ·
`Vulnerability` · `Integrity Violation` · `Resolution`.
See [`../ubiquitous-language.md`](../ubiquitous-language.md).

## Aggregates

### SecurityEvent
- **Root**: `SecurityEvent` (`security-event.model.ts`).
- **Identity**: `SecurityEventId`.
- **Fields**: `type`, `severity`, `userId?`, `ip?`, `userAgent?`,
  `correlationId`, `details`, `resolution: open | acknowledged |
  resolved | suppressed`, `resolvedAt?`, `resolvedBy?`,
  `resolutionNotes?`.
- **Invariants**:
  1. Once `resolution` reaches `resolved` or `suppressed`,
     transitions back are forbidden.
  2. `resolvedBy` is set iff `resolution ∈ {resolved, suppressed}`.
  3. `severity` is immutable after creation.
- **Notes**: Events generated by IAM (login failed, MFA failed) are
  also `SecurityEvent`s — IAM emits the domain event, the SecOps
  service materialises the aggregate.

### Finding (planned)
- **Root**: `Finding`.
- **Identity**: `FindingId`.
- **Fields**: `kind: secret | vulnerability | integrity | drift`,
  `severity`, `clusterId?`, `resourceRef?`, `evidenceRefs[]`,
  `status: open | acknowledged | resolved | suppressed`, `assignee?`,
  `firstSeenAt`, `lastSeenAt`, `occurrences` (counter for the same
  fingerprint).
- **Invariants**:
  1. Findings deduplicate by `(kind, fingerprint)` — the same
     issue across scans is one Finding with `lastSeenAt` updated.
  2. Resolution lifecycle as for `SecurityEvent`.
  3. `firstSeenAt <= lastSeenAt`.

## Value objects

- `Severity`: enum `info | low | medium | high | critical`. Total
  order; alerting thresholds defined in config.
- `Resolution` — value object capturing the resolution lifecycle and
  audit fields together.
- `Fingerprint` — sha256 over the canonical content of a finding,
  used for deduplication.

## Domain service

`SecurityService`:

- `recordEvent({type, severity, ...})` → persists `SecurityEvent`,
  returns the aggregate.
- `acknowledge(eventId, byUserId)`,
  `resolve(eventId, byUserId, notes)`,
  `suppress(eventId, byUserId, reason)` — lifecycle transitions.
- `scanSnapshotForSecrets(snapshotId)` — runs entropy + pattern
  detectors, emits `secops.FindingRaised` for each new
  fingerprint.
- `scanSnapshotForVulnerabilities(snapshotId)` — joins on a CVE
  feed (planned integration; mocked today).
- `monitorIntegrity(snapshotId, baselineId)` — diffs against a
  declared baseline.
- `bruteForceScore(userId, ip)` — derived metric from recent IAM
  events; consumed by IAM's lockout policy via service interface
  (cross-context call, no model coupling).

## Integration with neighbouring contexts

- **Identity & Access**: subscribes to login-failed events; can
  request session revocation via `IamService.revokeSession()` (not by
  writing to the user document).
- **Discovery**: consumes `discovery.SnapshotCompleted` to trigger
  scanners.
- **Compliance & Risk**: consumes `Finding` data as evidence for
  controls.
- **AI Intelligence**: consumes findings to summarise and recommend
  triage.
- **Audit**: every state transition is also an `AuditEvent`.

## Detection inventory

- **Secret detection**: regex + entropy detectors over discovered
  ConfigMaps, Deployments env, raw manifests.
- **Vulnerability scanning**: image SBOM × CVE database; planned
  integration.
- **Integrity monitoring**: hash-stable baselines vs. current
  snapshot.
- **Auth anomalies**: brute force, impossible travel (planned),
  device-fingerprint shifts.

## Severity taxonomy

| Severity   | Examples                                            |
| ---------- | --------------------------------------------------- |
| `critical` | Active exfil signature; valid AWS key in ConfigMap; admin login from new geo |
| `high`     | High-CVSS CVE on internet-exposed image; integrity drift on a privileged ConfigMap |
| `medium`   | Stale cert; low-entropy secret; non-default debug flag enabled |
| `low`      | Cosmetic drift; informational policy violation       |
| `info`     | Successful login from a new device                   |

Severity drives alert routing and SLA timers; the taxonomy lives in
`src/config/index.ts`.

## Out of scope (deliberately)

- Auto-remediation. SecOps surfaces findings; humans (or AI
  recommendations) act.
- Threat intelligence feed *aggregation* — we consume CVE/CVD feeds,
  not curate them.
- Network anomaly detection — different domain expertise; not in
  scope today.

## Open questions

- Whether to merge `SecurityEvent` and `Finding` into a single
  aggregate. Today they are separate because their lifecycles
  differ: events are immutable observations; findings dedupe and
  evolve.
- Suppression scope — per-cluster, per-namespace, global — needs
  product decision.
