# Bounded Context: Dashboard & Reporting

> *Supporting subdomain.* Aggregation and presentation; no business
> logic of its own.

## Purpose

Aggregate read-only views from every other context and render them as
dashboards (live, in-app) and reports (point-in-time, exportable).

## Ubiquitous language (canonical)

`Dashboard` · `Report`. See
[`../ubiquitous-language.md`](../ubiquitous-language.md).

## Source layout

| Concern        | File                                    |
| -------------- | --------------------------------------- |
| Domain service | `src/services/dashboard.service.ts`     |
| HTTP           | mounted via `src/app.ts` under `/api/v1/dashboard/*` |

Aggregate models for `DashboardDefinition` and `Report` are
**planned**; today's service composes JSON responses from upstream
service calls. Shapes below are the contract.

## Aggregates

### DashboardDefinition (planned)
- **Root**: `DashboardDefinition`.
- **Identity**: `DashboardId`.
- **Fields**: `name`, `ownerUserId`, `panels[]:
  { id, kind: chart | counter | list | ai_summary, source,
  filters }`, `sharedWithRoles[]`.
- **Invariants**:
  1. `panels[].source` references a known read API (allow-listed in
     code).
  2. Sharing visibility honours the viewer's permissions even if a
     dashboard is "public".

### Report (planned)
- **Root**: `Report`.
- **Identity**: `ReportId`.
- **Fields**: `kind: compliance | security | discovery | executive`,
  `format: json | csv | pdf`, `producedAt`, `producedBy: UserId`,
  `windowStart`, `windowEnd`, `payloadRef` (S3 / Mongo GridFS).
- **Invariants**:
  1. Immutable once produced.
  2. Permissions on the report are a snapshot of the producer's
     permissions at production time; downstream re-shares re-check.

## Value objects

- `Panel` — a renderable unit. Shape and `source` are validated
  against an allow-list at write time.
- `Window` — `{ start, end }` time range, validated for sanity
  (start ≤ end ≤ now).

## Domain service

`DashboardService`:

- `getOverview(viewerCtx)` — composes the default dashboard from
  Discovery (clusters, last-scan), SecOps (open findings),
  Compliance (latest score), Performance (live metrics), AI
  (latest executive summary).
- `getDashboard(dashboardId, viewerCtx)` — renders a saved
  dashboard, applying viewer permissions to each panel.
- `produceReport(input)` — gathers data from upstream services,
  renders, persists `Report`, emits no events (reports are passive).
- `listReports(filters)` and `getReport(reportId)`.

## Read-only relationship to everyone

Dashboard & Reporting **never writes** to another context's store.
It composes via:

- `IamService.getMe()` — viewer identity and permission summary.
- `DiscoveryService.listClusters()`, `getLatestSnapshot()`.
- `SecurityService.listFindings(filter)`, `getEvent(id)`.
- `ComplianceService.dashboard(frameworkId?)`, `getReport(id)`.
- `PerformanceService.getSummary()`, `getMetrics()`.
- `AIService.summariseExecutive(period)`.

Each call goes through the neighbouring context's service interface
and is bound to the *viewer's* permissions.

## Permission scoping

Every read in this context is filtered by the viewer's permissions:

- A panel referencing `Cluster X` is hidden if the viewer lacks
  `cluster:read` on that cluster's tenant scope.
- An AI summary is hidden if the viewer lacks
  `ai.analysis:read`.
- Mixing public and restricted panels in one dashboard is allowed;
  restricted panels render as "insufficient permissions" placeholders
  to the viewer.

## Caching

- Per-viewer short-TTL cache (e.g. 30s) for dashboard composition,
  keyed by `(viewerId, dashboardId, queryHash)`.
- Server-rendered PDFs are cached by `(reportId, format)` for the
  report's lifetime.
- Invalidation is event-driven: a `secops.FindingRaised` event
  invalidates the SecOps panel cache for any dashboard containing
  that source.

## Domain events

Dashboard & Reporting **does not publish** domain events. It
subscribes to others to invalidate caches.

## Integration

- **Frontend** (the dashboard SPA) is the primary consumer.
- **Customer integrations** that pull JSON exports use the same
  endpoints with API tokens.
- **Compliance & Risk** consumes `Report` indirectly — a compliance
  PDF is *generated by* Compliance and *served by* Dashboard.

## Out of scope (deliberately)

- **Authoring** complex dashboards via DSL — today, panels are
  configured via a small JSON schema; a future "dashboard builder"
  would not change this context's responsibilities.
- **Custom visualisations beyond the allow-listed kinds** — adding a
  new kind requires a code change and review (security: prevents
  arbitrary client-side scripting).
- **Long-running batch report jobs.** Reports today are produced
  synchronously up to a size cap; large exports are deferred to a
  job system once that exists.

## Open questions

- Whether to support saved cross-tenant dashboards once multi-tenancy
  hardens. Today: per-user, per-tenant scope.
- PDF rendering — wkhtmltopdf-style server vs. headless Chromium.
  Trade-off: dependency footprint vs. fidelity.
