# ADR-0007: Argon2id for password hashing

- **Status:** Accepted
- **Date:** 2026-05-08
- **Tags:** security, authentication

## Context

NOIP stores user credentials for the platform itself (operators,
auditors, admins). Even with MFA enforced, leaked password hashes are a
catastrophic event — they enable credential stuffing across other
systems. The hash function must:

1. Be memory-hard, raising the cost of GPU/ASIC cracking.
2. Have tunable parameters so we can re-cost it as hardware improves.
3. Be widely audited and have a maintained Node binding.

## Decision

We hash passwords with **Argon2id** via the `argon2` npm package
(`argon2 ^0.44.0`). Defaults follow the OWASP recommendation:

- `type: argon2id`
- `memoryCost: 65536` (64 MiB)
- `timeCost: 3`
- `parallelism: 1`
- per-user random 16-byte salt (generated by the library)

Hash format (PHC string) is stored in `User.password`. A pre-save
Mongoose hook in `src/models/user.model.ts` re-hashes whenever the field
is modified.

We rotate parameters by checking on login: if `argon2.needsRehash(hash,
options)` returns true, we re-hash the user's password with the new
parameters using the freshly-supplied plaintext.

## Alternatives considered

- **bcrypt** — battle-tested but capped at 72-byte passwords and not
  memory-hard. Adequate today, weaker tomorrow.
- **scrypt** — memory-hard but Argon2 is the more modern winner of the
  Password Hashing Competition and has better library support.
- **PBKDF2** — fast on GPUs; a regression from any of the above.

## Consequences

### Positive
- State-of-the-art resistance to offline cracking.
- `needsRehash` allows progressive parameter upgrades without forcing a
  password reset.
- PHC string stores the parameters with the hash — algorithm migration
  is seamless.

### Negative / costs
- Argon2 is a native binding; CI must install build tooling.
- 64 MiB per concurrent hash means login throughput is bounded by
  memory rather than CPU on small nodes.

### Risks and mitigations
- *Build-toolchain failures.* The Docker base image installs build deps
  in the `deps` stage; production image strips them.
- *DoS via login flood.* The `/auth/login` endpoint has a stricter rate
  limit (5 req / 15 min per IP) — see
  [ADR-0014](./0014-rate-limiting-redis-backed-sliding-window.md).
